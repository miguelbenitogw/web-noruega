# GA4 Implementation Plan (Detailed)

Date: 2026-03-16  
Project: `prueba-web-noruega`  
Measurement ID: `G-0X2BDYKVSP`

## 1. Objectives

1. Measure qualified leads generated by the website.
2. Understand which traffic sources produce the best leads.
3. Identify which pages and CTAs drive most contact form submissions.
4. Build a reliable weekly and monthly analytics workflow for decisions.

## 2. Scope

Included:
1. GA4 base setup and validation.
2. Event model for key user actions.
3. Conversion setup in GA4.
4. Reporting views and KPI definitions.
5. QA checklist and maintenance cadence.

Not included:
1. Server-side tracking / Measurement Protocol.
2. Full BI warehouse integration.
3. Advanced attribution modeling beyond GA4 defaults.

## 3. Tracking Architecture

Current status:
1. `VITE_GA_MEASUREMENT_ID` configured in [`.env`](c:\Users\PC\Desktop\prueba-web-noruega\.env).
2. GA script and events are initialized via [analytics.js](c:\Users\PC\Desktop\prueba-web-noruega\src\lib\analytics.js).
3. Contact form emits submit/error events from [Kontakt.jsx](c:\Users\PC\Desktop\prueba-web-noruega\src\components\Kontakt.jsx).

Implementation standard:
1. All custom events must use snake_case names.
2. Event params should be short, stable, and typed consistently.
3. No PII in event parameters (no raw email, no phone, no full name).

## 4. Event Taxonomy

## 4.1 Core Events (Phase 1)

1. `page_view` (automatic by GA4).
2. `contact_submit`
   - Params:
   - `contact_type` (`arbeidsgiver` | `kandidat`)
3. `contact_error`
   - Params:
   - `reason` (`missing_emailjs_config` | `send_failed` | other controlled values)
   - `contact_type`
4. `contact_blocked_bot`
   - Params:
   - `reason` (`honeypot` | `too_fast` | `rate_limit`)

## 4.2 Recommended Enhancement Events (Phase 2)

1. `cta_click`
   - Params:
   - `cta_id`
   - `section`
2. `nav_click`
   - Params:
   - `item`
3. `scroll_depth`
   - Params:
   - `percent` (25/50/75/90)
4. `faq_interaction`
   - Params:
   - `question_id`
   - `action` (`open` | `close`)

## 5. Conversion Strategy

Primary conversion:
1. `contact_submit` -> mark as conversion in GA4.

Secondary conversions:
1. `cta_click` on high-intent buttons (once implemented).
2. Optional: session-level engaged view thresholds (if needed later).

Rules:
1. Keep conversion count low and business-relevant.
2. Do not mark error/debug events as conversions.

## 6. KPI Framework

## 6.1 Acquisition KPIs

1. Users by channel (`Organic`, `Direct`, `Referral`, `Paid` if applicable).
2. Sessions by source/medium.
3. New users rate.

## 6.2 Engagement KPIs

1. Engagement rate.
2. Average engagement time per session.
3. Top landing pages by engaged sessions.

## 6.3 Lead KPIs (Main)

1. Total `contact_submit` events.
2. Conversion rate: `contact_submit / sessions`.
3. Leads by source/medium.
4. Leads by landing page.
5. Lead split by `contact_type`.

## 7. GA4 Admin Configuration Steps

1. Verify data stream is receiving traffic (`Realtime`).
2. Register custom events if needed (GA4 will detect automatically after first hits).
3. Mark `contact_submit` as conversion:
   - `Admin -> Data display -> Conversions -> New conversion event`.
4. Configure retention and internal traffic filters if applicable.
5. Optional: link Search Console and Google Ads (if in use).

## 8. QA and Validation Plan

## 8.1 Technical Validation

1. Browser Network check:
   - `googletagmanager.com/gtag/js?id=G-0X2BDYKVSP` returns 200.
   - `google-analytics.com/g/collect` returns 204.
2. Realtime view:
   - active user visible within 10-60 seconds.
3. Event validation:
   - Submit contact form and confirm `contact_submit`.
   - Force invalid state and confirm `contact_error`.
4. Cross-browser spot check:
   - Chrome, Edge, Safari (if available).

## 8.2 Data Quality Validation

1. Event names exactly match taxonomy.
2. Params appear in GA4 with expected values.
3. No PII in custom dimensions/params.
4. Conversion count aligns with actual contact submissions.

## 9. Reporting Setup

Weekly report:
1. Sessions by channel.
2. Leads (`contact_submit`) total and WoW trend.
3. Top 5 landing pages by conversion contribution.
4. Top source/medium by leads.
5. Errors (`contact_error`) trend.

Monthly report:
1. MoM lead growth.
2. Conversion rate trend.
3. Best performing acquisition channels.
4. Pages with highest and lowest lead efficiency.
5. Action backlog for CRO/SEO iteration.

## 11. Risks and Mitigations

1. Risk: Missing events after frontend changes.
   - Mitigation: QA checklist required before each deploy.
2. Risk: Wrong naming/params causing fragmented reporting.
   - Mitigation: Enforce event naming contract in code review.
3. Risk: Bot/spam inflating metrics.
   - Mitigation: Keep anti-bot controls active and monitor anomalies.
4. Risk: No clear owner for analytics.
   - Mitigation: Assign one owner for weekly health checks.

## 12. Acceptance Criteria

1. GA script loads on production with `G-0X2BDYKVSP`.
2. `contact_submit` appears in GA4 and is marked as conversion.
3. Realtime and standard reports show stable data for at least 7 days.
4. Weekly analytics review process is defined and running.
